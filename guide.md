# 通用项目上线后安全防护操作指南

版本：1.0  
适用对象：部署在云服务器、虚拟机、容器平台或托管平台上的 Web/API 项目  
适用阶段：上线前最后检查、正式上线、上线后持续运营  
目标读者：项目负责人、开发人员、运维人员、安全人员，以及受控执行任务的 AI 工程代理

---

## 1. 指南定位

本指南是一套Web/API项目生产安全操作基线。使用时应将文中的占位符替换为实际值：

```text
<DOMAIN>             正式主域名
<WWW_DOMAIN>         www 域名（如使用）
<SERVER_IP>          服务器公网 IP
<SSH_PORT>           SSH 管理端口
<APP_PORT>           应用本机监听端口
<APP_USER>           应用系统用户
<APP_SERVICE>        systemd 或容器服务名
<DATABASE_NAME>      数据库名称
<DATABASE_USER>      应用数据库角色
<BACKUP_LOCATION>    备份位置
<ALERT_RECIPIENT>    告警接收人
```

本指南不能保证系统“绝对安全”。它用于建立可验证、可恢复、可持续运营的纵深防御体系。

推荐安全等级：

| 等级 | 适用场景 | 要求 |
|---|---|---|
| 基础级 | 内部工具、低敏感度、少量用户 | 完成所有标记为 P0 的项目 |
| 生产级 | 正式商业系统、用户账号、业务数据 | 完成 P0 和 P1 |
| 高保障级 | 大量用户、敏感数据、关键业务 | 完成 P0、P1、P2，并安排独立安全测试 |

优先级定义：

- **P0**：上线阻断项。未完成时原则上不应正式上线。
- **P1**：生产必需项。应在上线前或上线后极短时间内完成。
- **P2**：高保障增强项。根据数据和业务风险实施。
- **条件项**：只有使用相关技术或业务能力时才适用。

## 2. 使用和执行规则

### 2.1 每项控制必须留下证据

不得仅以“已经配置”作为完成依据。每项至少保留一种证据：

- 配置文件或基础设施即代码变更。
- 命令输出。
- 自动化测试结果。
- 公网行为验证结果。
- 云平台控制台截图。
- 恢复演练记录。
- 告警测试通知。
- 第三方测试报告。

建议建立安全验收表：

| 控制编号 | 状态 | 执行人 | 完成时间 | 验证证据 | 回滚位置 | 备注 |
|---|---|---|---|---|---|---|
| SEC-001 | 未开始/进行中/已完成/不适用 |  |  |  |  |  |

### 2.2 任何生产变更都必须满足

- [ ] 修改前已确认当前服务健康。
- [ ] 已备份将要修改的配置。
- [ ] 已记录回滚命令或回滚目录。
- [ ] 配置先在非生产环境或离线环境验证。
- [ ] 数据库变更已评估锁表、耗时和回滚影响。
- [ ] 防火墙、SSH、DNS 和证书变更不会切断管理通道。
- [ ] 变更后执行健康检查、日志检查和公网验证。
- [ ] 没有把密码、密钥、令牌写入终端日志、Git 或聊天记录。

### 2.3 交给 AI 执行时的强制约束

AI 在开始写入前必须：

1. 只读审计当前状态。
2. 列出计划、风险、依赖和回滚方式。
3. 保护现有用户修改，不覆盖无关文件。
4. 不输出完整密码、密钥、Cookie、令牌或数据库连接串。
5. 不把秘密写入仓库。
6. 不关闭防火墙、审计、SELinux/AppArmor 等安全控制来“解决问题”。
7. 不执行不可逆数据库操作，除非有明确授权和可验证备份。
8. SSH、网络、安全组和 DNS 变更后必须验证新连接。
9. 每次只实施一个可回滚阶段。
10. 遇到权限不足时停止并说明所需权限，不得绕过权限边界。

## 3. 阶段一：资产、数据和风险识别

### SEC-001：建立资产清单（P0）

操作：

- [ ] 记录所有域名、子域名、IP、服务器、容器集群和云账号。
- [ ] 记录数据库、对象存储、缓存、消息队列和第三方服务。
- [ ] 记录代码仓库、CI/CD、镜像仓库和制品仓库。
- [ ] 记录所有管理员账号、服务账号、RAM/IAM 账号和 SSH 公钥。
- [ ] 记录证书、API Key、Webhook Secret、OAuth 凭据和数据库密钥的归属。
- [ ] 为每项资产指定负责人。
- [ ] 删除无主资产、测试实例、废弃子域和过期账号。

验收标准：

- 存在一份最新资产表。
- 任意生产资源都能找到负责人和用途。
- 不存在无人维护、无法解释的公网资源。

### SEC-002：数据分类（P0）

至少划分：

| 等级 | 示例 | 基本要求 |
|---|---|---|
| 公开 | 官网内容、公开文档 | 防篡改、可用性 |
| 内部 | 运营数据、内部配置 | 认证、授权、日志 |
| 敏感 | 手机号、邮箱、位置、订单 | 加密、最小权限、审计 |
| 高敏感 | 身份证、生物识别、财务、医疗 | 强认证、单独审批、严格保留和删除 |
| 秘密 | 密码、私钥、Token、备份密钥 | 不进入 Git、最小可见、轮换 |

操作：

- [ ] 列出收集的数据字段。
- [ ] 说明每个字段的收集目的。
- [ ] 删除没有业务必要的数据。
- [ ] 定义保存期限和删除机制。
- [ ] 标识是否涉及敏感个人信息。
- [ ] 标识数据是否会跨境传输。

验收标准：

- 每项数据都有目的、保留期限、访问角色和删除规则。
- 生产日志和分析平台不会无必要地记录敏感数据。

### SEC-003：威胁建模（P1）

至少回答：

- 谁可能攻击系统？
- 攻击者最想获得什么？
- 哪些入口暴露在公网？
- 哪些组件被攻破后影响最大？
- 哪些操作可以导致不可逆损失？
- 哪些第三方故障或密钥泄露会影响业务？

必须覆盖的威胁：

- 账号接管。
- 越权访问。
- SQL/命令/模板注入。
- XSS 和 CSRF。
- SSRF。
- 文件上传和路径穿越。
- 密钥泄露。
- 依赖供应链攻击。
- 数据库泄露或误删除。
- 云账号失控。
- DDoS 和资源耗尽。
- 勒索、备份破坏和日志清除。

验收标准：

- 对高风险威胁指定预防、检测和恢复措施。
- 关键威胁拥有可执行测试用例。

## 4. 阶段二：账号、云平台、域名和 DNS

### SEC-010：云主账号和管理员 MFA（P0）

操作：

- [ ] 云主账号启用 MFA。
- [ ] 域名注册商启用 MFA。
- [ ] DNS 托管平台启用 MFA。
- [ ] Git 托管、CI/CD、镜像仓库启用 MFA。
- [ ] 密码管理器启用 MFA。
- [ ] 优先使用 Passkey、FIDO2 安全密钥或其他抗钓鱼 MFA。
- [ ] 保存恢复码，并存放在独立安全位置。
- [ ] 禁止多人共用管理员账号。

验收标准：

- 实际登录测试会触发第二因素。
- 恢复设备丢失后仍有受控恢复流程。

### SEC-011：最小权限 IAM/RAM（P0）

操作：

- [ ] 日常操作不使用云主账号。
- [ ] 创建独立管理员和只读审计账号。
- [ ] 服务账号只获得完成任务所需权限。
- [ ] 删除长期不用的用户和权限。
- [ ] 禁止使用通配符管理员权限，除非有明确理由。
- [ ] AccessKey 设置负责人、用途、创建时间和轮换日期。
- [ ] 能使用实例角色、工作负载身份或短期凭据时，不使用长期 AccessKey。

验收标准：

- 能解释每个高权限账号的必要性。
- 不存在不明用途的长期高权限密钥。

### SEC-012：域名和 DNS 保护（P0）

操作：

- [ ] 域名开启注册商锁。
- [ ] 域名联系人邮箱和手机可用。
- [ ] DNS 账号启用 MFA。
- [ ] 删除废弃记录。
- [ ] 重要 DNS 变更启用审批或操作通知。
- [ ] 邮件域配置 SPF、DKIM 和 DMARC（条件项）。
- [ ] 评估 DNSSEC 支持并按需启用。
- [ ] 监控域名和证书到期时间。

验收标准：

- 域名不能被单一弱密码账号接管。
- 域名、DNS 和证书均有到期告警。

### SEC-013：云安全组（P0）

原则：默认拒绝，只开放明确需要的端口。

典型 Web 服务器入站：

| 端口 | 来源 | 用途 |
|---:|---|---|
| 80/TCP | 全网 | HTTP 跳转或证书验证 |
| 443/TCP | 全网 | HTTPS |
| SSH 管理端口 | 固定管理 IP/32 优先 | 运维 |

操作：

- [ ] 删除默认开放的数据库、缓存、远程桌面和管理端口。
- [ ] SSH 来源优先限制为固定 IP、VPN 或堡垒机。
- [ ] 数据库只允许应用所在私网或安全组访问。
- [ ] IPv4 和 IPv6 规则分别检查。
- [ ] 出站规则也按业务最小化，至少限制高风险管理协议。
- [ ] 安全组规则添加用途、负责人和到期时间。

验收标准：

- 公网端口扫描结果与设计一致。
- 服务器本机防火墙和云安全组形成双层控制。

## 5. 阶段三：秘密和密钥管理

### SEC-020：秘密不得进入 Git（P0）

秘密包括：

- 数据库密码。
- API Key。
- OAuth Client Secret。
- JWT 签名密钥。
- Cookie/Session 密钥。
- SSH 私钥。
- TLS 私钥。
- 云 AccessKey。
- Webhook Secret。
- 备份加密密钥。

操作：

- [ ] `.env`、密钥文件和本地配置已加入 `.gitignore`。
- [ ] 提交前运行秘密扫描。
- [ ] CI 对新提交执行秘密扫描。
- [ ] 示例配置只保留占位符。
- [ ] 日志、错误页和调试接口不输出秘密。

验收标准：

- 当前工作树和 Git 历史扫描均无有效秘密。

### SEC-021：已经进入 Git 的秘密必须轮换（P0）

仅从最新文件删除不够。正确步骤：

1. 立即在服务提供方创建新密钥。
2. 更新生产配置。
3. 重启或滚动重载服务。
4. 验证新进程使用新密钥。
5. 禁用旧密钥。
6. 检查日志和调用记录是否有异常使用。
7. 评估是否需要清理 Git 历史。

验收标准：

- 旧密钥已在服务提供方失效。
- 生产进程不再包含旧值。
- 新密钥受域名、IP、权限或配额限制。

### SEC-022：生产秘密存储（P1）

优先级：

1. 云 KMS/Secrets Manager。
2. 容器编排平台 Secret，并配合 KMS。
3. 权限严格的环境文件。
4. 不得把生产秘密写入镜像或前端代码。

操作：

- [ ] 秘密仅对运行账号可读。
- [ ] 文件权限通常不超过 `0640`，密钥优先 `0600`。
- [ ] 不在命令行参数中传递密码。
- [ ] 建立轮换周期。
- [ ] 轮换不会要求长时间停机。
- [ ] 密钥备份单独加密。

### SEC-023：前端 Key 的正确认识（P1）

浏览器必须使用的 Key 无法真正保密，因此必须：

- [ ] 设置域名白名单。
- [ ] 限制 API 权限。
- [ ] 设置配额和费用告警。
- [ ] 不让浏览器 Key 具备服务端管理权限。
- [ ] 高权限调用由后端代理完成。

## 6. 阶段四：服务器和操作系统加固

### SEC-030：最小化操作系统（P0）

操作：

- [ ] 使用仍在安全支持期内的发行版。
- [ ] 安装所有安全更新。
- [ ] 删除不需要的软件包、示例服务和测试工具。
- [ ] 禁止使用开发服务器直接对公网提供服务。
- [ ] 服务器时钟启用 NTP/Chrony。
- [ ] 设置正确时区并统一日志时间标准。
- [ ] 禁止普通用户随意读取生产配置。
- [ ] 定期检查异常 SUID/SGID 文件。

验收标准：

- 不存在已停止安全支持的操作系统。
- 没有无法解释的公网监听服务。

### SEC-031：自动安全更新（P1）

操作：

- [ ] 启用安全更新自动安装。
- [ ] 设置更新失败告警。
- [ ] 对需要重启的内核更新建立维护窗口。
- [ ] 更新前保证存在可恢复备份或快照。
- [ ] 重要更新先在预发布环境验证。

### SEC-032：系统用户隔离（P0）

操作：

- [ ] 应用使用独立、不可交互登录的系统用户。
- [ ] 应用不以 root 运行。
- [ ] 应用用户只拥有必要目录的读写权限。
- [ ] Web 根目录、代码目录和上传目录分离。
- [ ] 应用不能修改自身代码和依赖。
- [ ] 上传目录禁止执行程序。

验收标准：

- 应用被攻破后不能直接修改系统配置。
- 应用进程只能写入明确列出的目录。

### SEC-033：systemd 服务隔离（P1，Linux）

按兼容性测试以下能力：

```ini
NoNewPrivileges=true
PrivateTmp=true
PrivateDevices=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectControlGroups=true
ProtectKernelLogs=true
RestrictSUIDSGID=true
RestrictRealtime=true
RestrictNamespaces=true
LockPersonality=true
MemoryDenyWriteExecute=true
CapabilityBoundingSet=
AmbientCapabilities=
```

补充：

- [ ] 使用 `ReadWritePaths=` 只开放必要写目录。
- [ ] 使用 `RestrictAddressFamilies=` 限制协议族。
- [ ] 设置 `MemoryMax`、`TasksMax` 和 `LimitNOFILE`。
- [ ] 启用自动重启，但避免无限重启风暴。
- [ ] 运行 `systemd-analyze security <service>` 记录结果。

验收标准：

- 服务重启和主要功能测试通过。
- systemd 暴露评分达到可接受范围。

### SEC-034：内核和系统参数（P1）

按操作系统兼容性评估：

- [ ] 启用 ASLR。
- [ ] 限制内核地址和 dmesg 访问。
- [ ] 启用 ptrace 限制。
- [ ] 禁止非必要的非特权 BPF。
- [ ] 禁用 core dump。
- [ ] 禁止源路由、ICMP 重定向和发送重定向。
- [ ] 启用反向路径过滤。
- [ ] 启用 SYN Cookie。
- [ ] 禁用不使用的网络协议模块。

注意：

- 容器、网络代理、VPN、Kubernetes 节点可能依赖某些转发功能。
- 不得机械套用参数；修改后必须验证网络和容器功能。

## 7. 阶段五：SSH 和管理入口

### SEC-040：SSH 公钥化（P0）

操作：

- [ ] 禁止 root SSH 登录。
- [ ] 禁止密码登录。
- [ ] 禁止空密码。
- [ ] 仅允许明确的管理用户。
- [ ] 使用 ED25519、硬件密钥或其他现代密钥。
- [ ] 每位管理员使用独立公钥。
- [ ] 删除离职人员和废弃设备公钥。
- [ ] 私钥设置本地口令并由安全存储保护。

建议配置方向：

```text
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
AuthenticationMethods publickey
PermitEmptyPasswords no
AllowUsers <ADMIN_USER>
```

验收标准：

- 新终端可以通过公钥登录。
- 密码登录和 root 登录均失败。
- 不关闭当前 SSH 会话，直到新连接验证成功。

### SEC-041：限制 SSH 功能（P1）

若业务不需要：

```text
X11Forwarding no
AllowAgentForwarding no
AllowTcpForwarding no
AllowStreamLocalForwarding no
PermitTunnel no
PermitUserEnvironment no
DisableForwarding yes
```

同时设置：

- [ ] 登录宽限时间。
- [ ] 最大认证尝试次数。
- [ ] 最大会话数。
- [ ] 空闲连接清理。
- [ ] 详细安全日志。

### SEC-042：SSH 网络控制（P1）

推荐顺序：

1. VPN/零信任访问。
2. 堡垒机。
3. 安全组固定 IP 白名单。
4. 全网开放但仅公钥认证、Fail2ban 和严格限速。

修改 SSH 端口只能降低扫描噪声，不能替代公钥认证和网络限制。

## 8. 阶段六：本机防火墙和端口

### SEC-050：本机防火墙默认拒绝（P0）

操作：

- [ ] 入站默认拒绝。
- [ ] 出站按风险和业务能力控制。
- [ ] 只开放 HTTP、HTTPS 和受控管理端口。
- [ ] 数据库、缓存、应用内部端口只监听回环或私网。
- [ ] 邮件服务若只用于本机发送，仅监听回环。
- [ ] 检查 IPv6 规则。

验收标准：

- `ss`/`netstat` 监听结果与架构图一致。
- 外部端口扫描与安全组设计一致。
- 数据库和缓存不能从公网建立协议连接。

### SEC-051：端口和服务定期复核（P1）

每月执行：

- [ ] 列出所有 TCP/UDP 监听。
- [ ] 对每个监听进程确认用途和负责人。
- [ ] 停止无用服务。
- [ ] 检查容器端口是否意外映射到 `0.0.0.0`。
- [ ] 检查云负载均衡是否暴露管理接口。

## 9. 阶段七：反向代理、TLS 和 HTTP 边界

### SEC-060：统一使用 HTTPS（P0）

操作：

- [ ] HTTP 永久跳转到 HTTPS。
- [ ] 使用受信任证书。
- [ ] 证书私钥权限严格限制。
- [ ] 启用自动续期。
- [ ] 配置到期告警。
- [ ] 续期后自动重载反向代理。
- [ ] 至少支持 TLS 1.2 和 TLS 1.3。
- [ ] 禁用过时协议和弱密码套件。

验收标准：

- 浏览器无证书错误。
- 自动续期模拟或测试成功。
- 证书剩余不足 15 天时会产生告警。

### SEC-061：主域和未知 Host（P1）

操作：

- [ ] 选定唯一主域。
- [ ] `www` 或其他别名重定向到主域。
- [ ] 未知 HTTP Host 直接拒绝。
- [ ] 未知 HTTPS SNI 拒绝握手或返回默认拒绝页。
- [ ] 应用启用 Trusted Hosts 校验。

目的：

- 防止 Host Header 攻击。
- 防止错误域名共享生产站点。
- 降低缓存投毒和链接生成风险。

### SEC-062：安全响应头（P1）

建议基线：

```text
Strict-Transport-Security
Content-Security-Policy
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy
Cross-Origin-Opener-Policy
Cross-Origin-Resource-Policy
```

同时：

- [ ] 隐藏服务器版本。
- [ ] 隐藏上游框架服务器头。
- [ ] API 敏感响应使用 `Cache-Control: no-store`。
- [ ] 每个请求生成可追踪 Request ID。
- [ ] 反向代理和应用不要重复发送冲突的安全头。

### SEC-063：CSP（P1）

起始基线：

```text
default-src 'self';
script-src 'self';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self';
object-src 'none';
base-uri 'self';
frame-ancestors 'none';
form-action 'self';
upgrade-insecure-requests;
```

实施步骤：

1. 先使用 `Content-Security-Policy-Report-Only`。
2. 收集真实违规。
3. 按资源域名逐项白名单。
4. 不使用 `*` 解决问题。
5. 优先使用 nonce/hash，不使用脚本 `'unsafe-inline'`。
6. 第三方库要求 `'unsafe-eval'` 时记录风险接受和替代计划。
7. 对高风险第三方组件考虑隔离子域或 iframe 沙箱。

验收标准：

- 页面主要功能无 CSP 错误。
- CSP 不允许任意外域脚本。
- `object-src`、`base-uri`、`frame-ancestors` 和 `form-action` 已设置。

### SEC-064：限流和资源保护（P0）

分别设置：

- [ ] 登录接口限流。
- [ ] 注册、验证码、密码重置接口限流。
- [ ] 通用 API 限流。
- [ ] 上传接口大小和速率限制。
- [ ] 单 IP 并发连接限制。
- [ ] 请求头、请求体和读取超时。
- [ ] 后端工作进程和数据库连接池上限。
- [ ] 昂贵查询、导出和 AI 调用设置用户级配额。

验收标准：

- 恶意重复请求会收到 429 或被临时阻止。
- 正常用户不会被合理峰值误伤。

### SEC-065：反向代理信任链（P0）

操作：

- [ ] 后端仅信任真实反向代理的 `X-Forwarded-*`。
- [ ] 后端内部端口不直接暴露公网。
- [ ] 反向代理覆盖客户端伪造的转发头。
- [ ] 若使用 CDN/WAF，只信任官方出口 IP。
- [ ] 真实客户端 IP 解析链经过测试。

## 10. 阶段八：应用认证、会话和权限

### SEC-070：密码策略（P0）

建议：

- [ ] 最小长度至少 12，管理员至少 14。
- [ ] 允许密码管理器生成的长密码。
- [ ] 最大长度不低于 64，避免无上限造成 DoS。
- [ ] 检查常见泄露密码。
- [ ] 不强制无意义的周期改密，除非发生泄露或合规要求。
- [ ] 不截断密码。
- [ ] 密码哈希使用 Argon2id，或合理配置的 bcrypt/scrypt/PBKDF2。
- [ ] 每个密码使用独立随机盐。

禁止：

- 明文密码。
- MD5/SHA1/SHA256 直接哈希密码。
- 可逆加密存储密码。
- 在日志中记录密码。

### SEC-071：应用 MFA（P1；管理员 P0）

优先方案：

1. Passkey/WebAuthn。
2. TOTP。
3. 推送确认。
4. 短信仅作为较弱的补充方案。

必须设计：

- [ ] MFA 注册验证。
- [ ] 恢复码一次性显示和哈希存储。
- [ ] 设备丢失恢复流程。
- [ ] MFA 重置需要重新认证和审计。
- [ ] 高风险操作要求重新验证。
- [ ] 禁止客服或管理员无审计地绕过 MFA。

### SEC-072：登录防护（P0）

- [ ] 登录失败返回统一错误，不泄露用户是否存在。
- [ ] 限制 IP、账号和设备维度的尝试次数。
- [ ] 使用渐进延迟或临时锁定。
- [ ] 记录成功、失败、锁定和 MFA 事件。
- [ ] 新设备或异常位置登录通知用户。
- [ ] 管理员登录可以增加 IP 白名单。

### SEC-073：会话管理（P0）

Cookie 建议：

```text
Secure
HttpOnly
SameSite=Lax 或 Strict
Path=/
优先使用 __Host- 前缀
```

操作：

- [ ] 会话标识使用加密安全随机数。
- [ ] 登录后轮换会话 ID。
- [ ] 权限提升后轮换会话 ID。
- [ ] 设置空闲超时和绝对超时。
- [ ] 限制并展示活跃会话。
- [ ] 支持注销其他设备。
- [ ] 服务端注销立即失效。
- [ ] 密码修改、MFA 重置后使旧会话失效。

### SEC-074：CSRF（P0）

Cookie 会话系统必须：

- [ ] 对 POST/PUT/PATCH/DELETE 使用 CSRF Token。
- [ ] 验证 Origin/Referer 作为补充控制。
- [ ] 不使用 GET 修改状态。
- [ ] Token 与用户会话绑定。
- [ ] 登录、注销和密码修改也纳入评估。
- [ ] SameSite 不能作为唯一 CSRF 防线。

### SEC-075：授权和越权（P0）

原则：

- 每次请求都在服务端授权。
- 前端隐藏按钮不等于权限控制。
- 默认拒绝。
- 资源所有权必须由服务端验证。

测试：

- [ ] 普通用户不能访问管理员 API。
- [ ] 用户 A 不能读取或修改用户 B 的资源。
- [ ] 修改 URL、ID、请求体不能越权。
- [ ] 批量接口逐项授权。
- [ ] 导出接口按数据范围授权。
- [ ] WebSocket、文件下载和后台任务同样授权。

多租户系统额外要求：

- [ ] 每个查询强制租户边界。
- [ ] 数据库层可按风险启用 RLS。
- [ ] 缓存 Key 包含租户标识。
- [ ] 对象存储路径不能跨租户访问。

### SEC-076：账号生命周期（P1）

- [ ] 注册需要验证邮箱或手机。
- [ ] 密码重置 Token 一次性、短时、不可预测。
- [ ] 修改邮箱、手机、密码和 MFA 时重新认证。
- [ ] 删除账号有明确数据处理流程。
- [ ] 禁用账号立即终止会话。
- [ ] 定期清理长期未使用账号。
- [ ] 管理员创建、授权、禁用均有审计记录。

## 11. 阶段九：输入、输出和常见 Web 漏洞

### SEC-080：统一输入验证（P0）

操作：

- [ ] 服务端验证所有输入。
- [ ] 使用明确类型、长度、范围、枚举和格式。
- [ ] 拒绝未知字段或谨慎处理。
- [ ] 分页大小设置上限。
- [ ] 排序字段使用白名单。
- [ ] 文件名、URL、路径和正则表达式单独验证。
- [ ] 反序列化只接受明确结构。

### SEC-081：SQL/NoSQL 注入（P0）

- [ ] 使用 ORM 或参数化查询。
- [ ] 不拼接用户输入到 SQL。
- [ ] 表名、列名和排序字段使用白名单。
- [ ] 数据库账号不使用超级用户。
- [ ] 数据库权限按表和操作最小化。
- [ ] 对原生 SQL 进行专项审查。

### SEC-082：命令注入（P0）

- [ ] 能使用库函数时不调用 Shell。
- [ ] 子进程使用参数数组。
- [ ] 禁止 `shell=true`，除非有严格理由。
- [ ] 可执行文件使用固定绝对路径。
- [ ] 用户输入不得决定命令名或参数结构。
- [ ] 设置超时、工作目录和受限环境变量。

### SEC-083：XSS（P0）

- [ ] 默认使用框架转义输出。
- [ ] 禁止把不可信内容传入 `innerHTML`、`v-html` 等接口。
- [ ] 必须渲染 HTML 时使用成熟白名单清洗器。
- [ ] URL、CSS、HTML 属性和 JavaScript 上下文分别编码。
- [ ] 富文本上传和展示单独审查。
- [ ] 使用 CSP 作为补充，不把 CSP 当成唯一防线。
- [ ] 对 DOM XSS 执行前端静态检查。

### SEC-084：SSRF（P0）

任何“抓取 URL、图片代理、Webhook、导入链接、PDF 生成、预览网页”功能都必须：

- [ ] 只允许 HTTP/HTTPS。
- [ ] 使用域名或目标白名单。
- [ ] 解析 DNS 后阻止回环、私网、链路本地和云元数据地址。
- [ ] 每次重定向后重新验证目标。
- [ ] 限制端口。
- [ ] 设置连接、读取、总时长和响应体上限。
- [ ] 使用独立低权限出站网络。
- [ ] 阻止访问云元数据服务。

### SEC-085：路径穿越和文件读取（P0）

- [ ] 用户输入不得直接拼接文件系统路径。
- [ ] 使用随机内部文件名。
- [ ] `resolve` 后验证路径仍在允许根目录内。
- [ ] 禁止读取 `.env`、密钥、数据库、日志和系统文件。
- [ ] 下载接口验证资源所有权。
- [ ] Nginx/对象存储不暴露目录索引。

### SEC-086：模板和表达式注入（P1）

- [ ] 用户内容不得作为服务端模板代码执行。
- [ ] 禁止不可信表达式进入 `eval`、模板引擎和规则引擎。
- [ ] 沙箱不能替代严格输入模型。
- [ ] AI 生成内容不能直接作为代码、SQL、Shell 或模板执行。

## 12. 阶段十：文件上传和对象存储

### SEC-090：上传安全（P0）

操作：

- [ ] 设置单文件和总请求大小。
- [ ] 按文件魔数检测真实类型。
- [ ] 扩展名使用白名单。
- [ ] 拒绝可执行文件、脚本和危险压缩包。
- [ ] 文件名由服务端随机生成。
- [ ] 上传目录与代码目录分离。
- [ ] 上传目录不可执行。
- [ ] 图片按需重新编码以移除恶意内容和元数据。
- [ ] 文档类文件按风险使用杀毒或沙箱扫描。
- [ ] 压缩包限制文件数、解压大小和路径，防止 Zip Slip/Zip Bomb。
- [ ] 私有文件必须经过授权下载。

### SEC-091：对象存储（P0）

- [ ] Bucket 默认私有。
- [ ] 禁止公共写入。
- [ ] 按前缀和角色设置最小权限。
- [ ] 使用短期签名 URL。
- [ ] 设置 CORS 白名单。
- [ ] 开启服务端加密。
- [ ] 开启访问日志。
- [ ] 重要备份启用版本控制和对象锁。
- [ ] 防止应用凭据拥有删除所有备份的权限。

## 13. 阶段十一：API、Webhook 和第三方服务

### SEC-100：API 基线（P0）

- [ ] 使用 HTTPS。
- [ ] 所有非公开 API 需要认证。
- [ ] 每个接口执行授权。
- [ ] 请求和响应大小有上限。
- [ ] 分页和导出有上限。
- [ ] 错误响应不泄露堆栈、SQL、路径或秘密。
- [ ] API 文档在生产环境关闭或受认证保护。
- [ ] 废弃 API 有下线时间。
- [ ] 不在 URL 查询参数中放敏感 Token。

### SEC-101：CORS（P0）

- [ ] 只允许明确的正式域名。
- [ ] 使用凭据时禁止 `Access-Control-Allow-Origin: *`。
- [ ] 只开放必要方法和请求头。
- [ ] 开发域名不能进入生产配置。
- [ ] CORS 不是认证或授权机制。

### SEC-102：Webhook（P0，条件项）

- [ ] 使用 HMAC 或非对称签名验证。
- [ ] 签名覆盖原始请求体和时间戳。
- [ ] 拒绝过期时间戳。
- [ ] 使用事件 ID 防止重放。
- [ ] 返回前后均确保幂等。
- [ ] Webhook Secret 可轮换。
- [ ] 限制请求体和处理时间。
- [ ] 失败重试不会重复执行危险操作。

### SEC-103：OAuth/OIDC（P1，条件项）

- [ ] 使用授权码流程和 PKCE。
- [ ] 精确匹配 Redirect URI。
- [ ] 验证 state、nonce、issuer、audience 和签名。
- [ ] 不接受 `alg=none`。
- [ ] Access Token 短时有效。
- [ ] Refresh Token 轮换并可撤销。
- [ ] 第三方登录账号绑定流程防止账号合并劫持。

### SEC-104：第三方服务治理（P1）

- [ ] 记录每个第三方处理的数据。
- [ ] 评估服务条款、隐私和数据存储区域。
- [ ] Key 设置最小权限、域名/IP 白名单和配额。
- [ ] 配置费用异常告警。
- [ ] 第三方故障时具备降级策略。
- [ ] 第三方密钥泄露有轮换流程。

## 14. 阶段十二：数据库和缓存

### SEC-110：数据库网络隔离（P0）

- [ ] 数据库不直接暴露公网。
- [ ] 优先使用 Unix Socket、私网或安全组。
- [ ] 管理连接通过 VPN、堡垒机或临时隧道。
- [ ] 数据库 TLS 按网络和合规要求启用。
- [ ] 默认数据库和测试账号已删除或禁用。

### SEC-111：数据库角色最小权限（P0）

建议角色分离：

| 角色 | 权限 |
|---|---|
| 数据库所有者 | 拥有对象，不用于应用登录 |
| 迁移角色 | 执行结构变更，仅部署时使用 |
| 应用运行角色 | 只执行日常读写 |
| 只读分析角色 | 只读必要表或视图 |
| 备份角色 | 只具备备份所需能力 |

操作：

- [ ] 应用角色不是超级用户。
- [ ] 禁止应用角色创建数据库或角色。
- [ ] 设置连接数上限。
- [ ] 对审计表禁止更新和删除。
- [ ] 迁移凭据不常驻应用环境。

### SEC-112：数据库超时和资源限制（P1）

- [ ] 查询超时。
- [ ] 锁等待超时。
- [ ] 空闲事务超时。
- [ ] 连接池上限。
- [ ] 慢查询日志。
- [ ] 大查询、批处理和导出使用单独队列或资源池。

### SEC-113：数据库审计（P1）

记录：

- 登录和失败登录。
- 权限变更。
- 角色创建和删除。
- 结构变更。
- 高风险表访问。
- 备份和恢复。
- 超时、死锁和异常大量查询。

日志不得记录：

- 密码。
- 完整 Token。
- 不必要的敏感字段。

### SEC-114：缓存安全（P0，条件项）

- [ ] Redis/Memcached 不开放公网。
- [ ] 使用认证和 TLS（跨主机时）。
- [ ] 禁用或重命名高风险管理命令（按实现评估）。
- [ ] 设置内存上限和淘汰策略。
- [ ] 缓存 Key 包含用户/租户边界。
- [ ] 敏感信息不长期存入缓存。
- [ ] Session 缓存具备过期和撤销机制。

## 15. 阶段十三：依赖、构建和供应链

### SEC-120：依赖锁定（P0）

- [ ] 使用 lock 文件或固定版本。
- [ ] 构建使用确定性依赖解析。
- [ ] 不直接依赖未知来源压缩包或脚本。
- [ ] 不从不可信镜像安装生产依赖。
- [ ] 定期检查依赖维护状态和许可证。

### SEC-121：依赖漏洞扫描（P0）

CI 至少执行：

- 前端依赖审计。
- 后端依赖审计。
- 容器镜像扫描。
- 操作系统包扫描。
- Git 秘密扫描。
- 静态安全扫描。

处理规则：

- 严重/高危且可利用：阻断发布。
- 中危：评估可利用性和修复计划。
- 无修复版本：实施补偿控制并记录风险接受。

### SEC-122：CI/CD 安全（P0）

- [ ] CI 账号启用 MFA。
- [ ] 合并前需要评审。
- [ ] 保护主分支和发布标签。
- [ ] CI 使用最小权限 Token。
- [ ] Fork/PR 代码不能读取生产 Secret。
- [ ] 生产部署需要审批。
- [ ] 构建日志对秘密脱敏。
- [ ] 制品生成校验和。
- [ ] 部署使用经过测试的同一制品，不在生产服务器临时构建。

### SEC-123：制品和发布完整性（P1）

- [ ] 构建产物具有 SHA-256。
- [ ] 条件允许时签名镜像和制品。
- [ ] 生产服务器只接收受信任制品。
- [ ] 保留发布版本和部署时间。
- [ ] 当前版本通过只读软链接或明确版本目录指向。
- [ ] 能快速切回上一版本。

### SEC-124：SBOM（P2）

- [ ] 为每次发布生成软件物料清单。
- [ ] 记录直接和间接依赖。
- [ ] 关联版本、许可证和已知漏洞。
- [ ] 保留历史 SBOM，支持漏洞追溯。

## 16. 阶段十四：容器和编排平台

### SEC-130：容器镜像（条件项，P0）

- [ ] 删除不必要 Linux capabilities。

### SEC-131：容器运行时（条件项，P1）

- [ ] 不使用 privileged。
- [ ] 不挂载 Docker Socket。
- [ ] 不挂载宿主机根目录。
- [ ] 使用 seccomp、AppArmor 或 SELinux。

- [ ] 数据库和缓存不映射公网端口。
- [ ] 健康检查不能泄露敏感信息。

- [ ] 禁止匿名和默认 ServiceAccount 滥用。

- [ ] Ingress、证书和公网负载均衡定期审查。

## 17. 阶段十五：日志、审计、监控和告警

### SEC-140：应用安全日志（P0）

必须记录：

- 登录成功和失败。
- MFA 注册、验证和重置。
- 密码修改和重置。
- 会话创建、注销和撤销。
- 管理员操作。
- 权限和角色变更。
- 敏感数据导出。
- 关键配置变更。
- 上传、删除和批量操作。
- Webhook 验证失败。

每条日志建议包含：

- 时间。
- Request ID。
- 事件类型。
- 用户和租户标识。
- 来源 IP。
- User-Agent 摘要。
- 结果。
- 目标资源。

禁止记录：

- 密码。
- 完整 Session/Cookie。
- 完整 Access Token。
- 信用卡安全码。
- 不必要的身份证、手机号等完整敏感数据。

### SEC-141：主机和基础设施日志（P1）

- [ ] SSH 日志。
- [ ] sudo 日志。
- [ ] 防火墙日志。
- [ ] Fail2ban/入侵防护日志。
- [ ] Nginx 访问和错误日志。
- [ ] 数据库审计日志。
- [ ] 云控制台操作日志。
- [ ] DNS 和证书变更记录。
- [ ] systemd 服务异常。

### SEC-142：日志保护（P1）

- [ ] 日志权限最小化。
- [ ] 日志轮换和磁盘上限。
- [ ] 关键日志发送到外部或集中平台。
- [ ] 时间同步。
- [ ] 防止应用用户删除审计日志。
- [ ] 定义保留期限。
- [ ] 对敏感日志加密。

### SEC-143：监控指标（P0）

至少监控：

- CPU。
- 内存。
- 磁盘空间和 inode。
- 网络流量。
- 应用健康。
- 5xx 比例。
- 登录失败率。
- 请求延迟。
- 数据库连接数。
- 慢查询和锁等待。
- 队列积压。
- 证书剩余时间。
- 备份新鲜度。
- 容器重启次数。

### SEC-144：外部告警（P0）

告警不能只写本机日志，必须通过至少一种独立渠道发送：

- 邮件。
- 短信。
- 电话。
- 企业微信/钉钉/Slack。
- PagerDuty 等值班平台。

必须告警：

- 网站连续不可用。
- 5xx 激增。
- 登录爆破或异常管理员登录。
- 备份失败或超期。
- 证书即将过期。
- 磁盘超过阈值。
- 审计服务停止。
- 完整性检查发现异常。
- 数据库不可用。
- 云账号和安全组变更。
- 费用异常增长。

验收标准：

- 实际触发一次测试告警。
- 接收人确认可以收到和处理。
- 告警包含系统、时间、严重度和处理链接。

### SEC-145：文件完整性和主机审计（P1）

- [ ] 启用 auditd 或等效审计。
- [ ] 监控身份文件、SSH、sudo、防火墙、反向代理、数据库和服务配置。
- [ ] 监控代码发布目录和 Web 根目录。
- [ ] 使用 AIDE 或等效工具建立完整性基线。
- [ ] 基线更新需要审批。
- [ ] 发现变化后区分合法发布和异常篡改。

## 18. 阶段十六：备份、恢复和灾难恢复

### SEC-150：备份策略（P0）

至少采用 3-2-1 思路：

- 至少 3 份数据。
- 至少 2 种存储介质或位置。
- 至少 1 份异地或离线副本。

必须备份：

- 数据库。
- 用户上传文件。
- 必要业务文件。
- 基础设施配置。
- 应用环境配置。
- 密钥恢复材料。
- DNS、证书和第三方配置清单。

代码通常由 Git 和构建制品恢复，不应依赖生产服务器副本。

### SEC-151：备份加密（P0）

- [ ] 备份静态加密。
- [ ] 密钥与备份分离存放。
- [ ] 传输过程使用加密通道。
- [ ] 备份文件生成 SHA-256。
- [ ] 备份日志不输出密钥。
- [ ] 备份账号不能修改生产数据。

### SEC-152：不可变和防删除（P1）

- [ ] 异地存储启用版本控制。
- [ ] 启用对象锁/WORM 或保留策略。
- [ ] 应用服务器凭据不能删除所有历史备份。
- [ ] 删除备份需要独立管理员或延迟生效。
- [ ] 云账号失陷时仍保留恢复路径。

### SEC-153：备份计划（P0）

示例：

- 数据库 WAL/增量：持续或高频。
- 数据库差异：每日。
- 数据库全量：每周。
- 上传文件：每日或按业务 RPO。
- 配置和密钥恢复材料：变更后立即。
- 保留：按业务、法规和存储成本确定。

必须定义：

- RPO：最多允许丢失多少数据。
- RTO：最多允许多久恢复服务。

### SEC-154：真实恢复演练（P0）

仅有备份文件不代表可以恢复。

至少每季度：

1. 选择最新备份。
2. 验证校验和。
3. 解密到受控临时位置。
4. 创建隔离测试数据库或环境。
5. 完整恢复。
6. 查询关键业务数据。
7. 验证上传文件可读。
8. 记录耗时和问题。
9. 删除临时明文。
10. 更新恢复文档。

验收标准：

- 恢复演练满足 RPO/RTO。
- 恢复步骤不依赖某一位人员的记忆。

### SEC-162：数据最小化（P0）

### SEC-163：跨境数据（条件项，P0）

- [ ] 评估适用的安全评估、标准合同或认证路径。
- [ ] 完成必要的个人信息保护影响评估。
- [ ] 只传输实现目的所需的最少数据。
- [ ] 保存合规材料和处理记录。

- [ ] 迁移在相同数据库版本上测试。

- [ ] 数据库连接。

- [ ] 第三方服务。
- [ ] HTTPS、证书和安全响应头。
- [ ] 主域重定向。
- [ ] 未知 Host/SNI 拒绝。
- [ ] 端口监听。
- [ ] 日志无秘密和异常堆栈。
- [ ] 新进程确实加载新配置。
- [ ] 监控和告警恢复正常。

## 22. 阶段二十：应急响应

### SEC-180：建立应急响应计划（P0）

计划必须包含：

- 联系人和决策人。
- 云平台、域名、DNS、代码仓库和第三方联系方式。
- 事件严重度。
- 隔离、取证、恢复和通知流程。
- 法律和监管报告要求。
- 对外沟通负责人。

严重度示例：

| 等级 | 示例 | 响应 |
|---|---|---|
| S1 | 核心数据泄露、云账号接管、勒索 | 立即响应，最高负责人参与 |
| S2 | 管理员账号异常、持续入侵、关键服务中断 | 30 分钟内响应 |
| S3 | 扫描、单个漏洞、非核心服务异常 | 当日处理 |
| S4 | 一般告警和配置偏差 | 按计划修复 |

### SEC-181：密钥泄露处置

1. 不等待调查结束，先轮换。
2. 禁用旧密钥。
3. 检查调用、费用和访问日志。
4. 确认新进程加载新密钥。
5. 清理仓库、制品、日志和聊天中的副本。
6. 评估受影响数据和通知义务。
7. 修复导致泄露的流程。

### SEC-182：服务器失陷处置

1. 保留证据，不立即清除所有日志。
2. 从网络隔离受影响主机。
3. 撤销主机可访问的所有密钥。
4. 从可信镜像重建，不在原主机“就地清毒”后继续运行。
5. 从已验证备份恢复。
6. 对数据库、对象存储和云操作日志进行调查。
7. 强制失效相关会话。
8. 完成根因分析。

### SEC-183：数据库误删或损坏

1. 停止继续写入。
2. 记录故障时间。
3. 选择满足 RPO 的全量、差异和 WAL。
4. 在隔离环境恢复并验证。
5. 明确数据缺口后再切换。
6. 保留故障数据库用于调查。

### SEC-184：DDoS 和流量攻击

- [ ] 启用云平台基础 DDoS 防护。
- [ ] 准备 CDN/WAF/高防切换方案。
- [ ] 对昂贵接口设置配额。
- [ ] 降级非核心功能。
- [ ] 缩短静态内容链路。
- [ ] 与云厂商建立紧急联系路径。

## 24. 交给 AI 的主执行提示词

可将以下模板复制给具备服务器和代码访问权限的 AI。必须替换占位符，并按阶段执行，不要一次性授权所有高风险变更。

```text
你负责对当前项目执行生产安全加固。

目标：
按照《通用项目上线后安全防护操作指南》逐项审计、实施、验证并记录证据。

环境：
- 仓库：<REPOSITORY_PATH>
- 正式域名：<DOMAIN>
- 服务器：<SERVER_ALIAS_OR_IP>
- 技术栈：<STACK>
- 数据库：<DATABASE>
- 云平台：<CLOUD_PROVIDER>

强制规则：
1. 先只读审计，不要立即修改。
2. 输出当前状态、风险、实施计划、依赖和回滚方式。
3. 保护现有修改，不覆盖无关文件。
4. 不在回答、日志、Git 或脚本中暴露完整秘密。
5. 每个阶段修改前创建回滚副本。
6. 使用配置语法检查、测试和公网行为验证。
7. SSH、防火墙、安全组、DNS 和数据库修改必须避免锁死管理通道。
8. 数据库不可逆操作需要明确授权和已验证备份。
9. 不通过关闭安全控制解决兼容问题。
10. 发现需要云控制台、MFA 注册、法律判断或第三方账号权限时，停止该项并给出准确操作步骤。
11. 每完成一项，报告控制编号、变更、验证证据、剩余风险和回滚位置。
12. 未验证成功的项目不得标记为完成。

执行顺序：
A. 资产、端口和账号只读审计
B. 代码、依赖、秘密和认证安全
C. 数据库权限、网络和审计
D. 反向代理、TLS、CSP、限流和安全头
E. SSH、防火墙、systemd、自动更新、auditd、Fail2ban、AIDE
F. 加密备份和真实恢复演练
G. 日志、监控和外部告警
H. 公网验证、测试和最终报告

先开始 A 阶段，只读检查完成后再决定是否进入写入阶段。
```

## 25. AI 分阶段任务模板

### 模板 A：只读安全审计

```text
只进行只读检查，不修改文件、服务、数据库、DNS 或云资源。

检查：
- 仓库状态、技术栈、依赖和秘密暴露
- 公网域名、TLS、安全响应头和未知 Host
- 服务器监听端口、服务、用户和权限
- SSH、防火墙、自动更新、日志和审计
- 数据库网络、角色、权限、审计和备份
- 云安全组、IAM/MFA、DNS 和告警（如有只读权限）

输出：
1. 当前架构
2. 风险清单，按 P0/P1/P2 排序
3. 可直接实施项
4. 需要人工或云控制台处理项
5. 每项验证方法
6. 不要输出任何完整秘密
```

### 模板 B：代码和认证加固

```text
对代码执行安全加固，范围仅限：
- 密码哈希和密码策略
- 登录限流和审计
- 安全 Cookie、会话超时和会话撤销
- CSRF、CORS、Trusted Hosts
- 服务端授权和越权测试
- 输入长度、类型和枚举限制
- SQL、命令、XSS、SSRF、路径和上传安全
- 生产调试功能关闭

要求：
- 先运行现有测试
- 修改后增加安全测试
- 执行依赖审计和静态扫描
- 不改变无关业务行为
- 输出文件、测试结果和剩余风险
```

### 模板 C：服务器加固

```text
对生产服务器执行分阶段、可回滚加固。

范围：
- 非 root 应用用户
- systemd/容器隔离
- SSH 公钥、禁止 root 和密码
- 云安全组与本机防火墙最小开放
- Nginx/TLS/安全头/未知 Host/限流
- 自动安全更新
- Fail2ban、auditd、AIDE
- 内核和 core dump 安全参数

要求：
- 保持现有 SSH 会话
- SSH 修改后先验证新连接
- Nginx 修改前后运行语法检查
- 每一步保留回滚副本
- 不重启数据库，除非确有必要并已说明影响
- 逐项报告验证结果
```

### 模板 D：数据库加固

```text
审计并加固数据库。

目标：
- 禁止公网访问
- 所有者、迁移、应用、只读、备份角色分离
- 应用角色非超级用户且最小权限
- 设置连接、查询、锁和空闲事务超时
- 启用连接、权限变更、DDL、慢查询和必要审计
- 审计表禁止应用更新和删除
- 健康检查验证数据库连接

要求：
- 修改前导出角色、权限和配置
- 不改变业务数据
- 不打印密码
- 用测试查询验证应用仍可正常工作
- 输出回滚 SQL 和证据
```

### 模板 E：备份和恢复

```text
建立生产备份和恢复体系。

要求：
- 数据库全量、差异/增量和日志归档
- 上传文件备份
- AES-256 或云 KMS 加密
- 密钥与备份分离
- SHA-256 校验
- 本地保留和异地副本
- 定时器和失败告警
- 创建隔离测试数据库执行真实恢复
- 查询核心表验证数据
- 删除临时明文和测试数据库
- 记录 RPO、RTO、恢复耗时和完整步骤
```

### 模板 F：最终验收

```text
只做验证和报告，不新增高风险变更。

验证：
- 首页、健康检查、关键 API
- HTTPS、证书、重定向、安全头和 CSP
- 未知 Host/SNI
- 公网端口和服务器监听
- SSH 新连接
- 核心服务状态
- 数据库不暴露公网
- 自动任务和最近一次结果
- Fail2ban、auditd、AIDE
- 备份新鲜度和恢复记录
- 前后端测试、依赖审计、静态扫描和生产构建
- 仓库无秘密

输出：
- 综合评分
- 已完成控制
- 未完成控制
- 风险接受项
- 证据和日期
- 未来 30/90 天行动计划
```

## 26. 验收评分方法

建议使用 100 分制：

| 领域 | 权重 |
|---|---:|
| 应用认证、会话和权限 | 20 |
| 输入、输出和业务逻辑安全 | 15 |
| 网络、TLS 和反向代理 | 15 |
| 主机、SSH 和运行时 | 12 |
| 数据库和存储 | 12 |
| 备份和恢复 | 10 |
| 日志、监控和告警 | 8 |
| 云账号、密钥和供应链 | 8 |

扣分原则：

- P0 未完成：每项扣 5～15 分，严重时判定不能上线。
- P1 未完成：每项扣 2～5 分。
- P2 未完成：每项扣 0.5～2 分。
- 仅有配置、没有验证证据：按未完成或只完成 50% 计算。
- 有备份但没有恢复演练：备份项最高只能获得一半分数。
- 有 MFA 但恢复流程可以绕过：MFA 项不能判定完成。

等级建议：

| 分数 | 等级 | 解释 |
|---:|---|---|
| 95～100 | A | 高保障生产级 |
| 90～94 | A- | 成熟的中高强度生产级 |
| 80～89 | B | 合格生产级，仍有明显增强项 |
| 70～79 | C | 基础防护，重要系统不建议直接上线 |
| 60～69 | D | 高风险，需优先整改 |
| 低于 60 | F | 不具备安全上线条件 |

评分不能替代渗透测试、合规审查或等级保护测评。
